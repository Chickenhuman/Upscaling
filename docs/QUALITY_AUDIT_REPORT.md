# 카드 카페 루프 품질 감사 보고서

작성일: 2026-04-24  
대상 커밋: `c16a7f4 feat: add card cafe loop web game`  
판정: 현재 빌드는 "출시 후보"라기보다 기능 시연용 MVP에 가깝다. 핵심 루프는 존재하지만, 캐릭터 미연시/덱빌딩 로그라이크로서 기대되는 아트, 연출, 튜토리얼, 현지화, 콘텐츠 밀도, 시스템 깊이가 부족하다.

정책 정정: 이후 README 기준이 확정되어, 성인 캐릭터 간의 명시적 성적 콘텐츠는 일정 수위까지 허용한다. 다만 15세 이상 이용 기준과의 충돌 가능성은 출시 전 별도 재검토가 필요하다.

## 감사 범위

- 정적 파일 구조, 이미지 에셋, UI/CSS, 게임 상태 머신, 카드/스토리 데이터, 문서, 검증 스크립트를 코드 기준으로 확인했다.
- 실제 브라우저 장시간 플레이 테스트나 다중 브라우저 수동 QA는 수행하지 않았다.
- `npm run check`는 통과하지만, 해당 검증은 문법/데이터 수량/시뮬레이션 위주이며 완성도 문제를 거의 잡지 못한다.

## 요약

- 출시 차단급 문제: 개별 캐릭터 일러스트 부재, 튜토리얼 부재, 한영 혼용, 애니메이션/피드백 부재, 문서와 실제 구현 불일치.
- 큰 구조적 문제: "스토리 125p/대사 125p"는 사실상 5개 템플릿을 반복 생성한 값이라 실제 서사 분량으로 보기 어렵다.
- 게임성 문제: 덱빌딩은 카드 사용/보상/전투 수치만 있고 대상 지정, 상태 이상, 적 패턴, 강화/제거, 상점 선택, 루트별 카드 해금 같은 핵심 깊이가 없다.
- 미연시 문제: 캐릭터별 고유 표정, 포즈, 이벤트 CG, 관계 단계별 대사, 루트 전용 갈등/해소 구조가 없다.
- 운영 문제: GitHub Pages 배포 설정, CI, 브라우저 자동 테스트, 접근성 검증, 저장 데이터 마이그레이션 전략이 없다.

## 출시 차단급 문제

### P0-1. 개별 캐릭터 일러스트가 없다

현재 `assets/images`에는 캐릭터 전용 파일이 없고 `characters.png` 한 장만 있다. `src/styles.css:379-384`는 모든 캐릭터와 대화 초상화를 같은 합성 이미지에서 `background-position`으로 잘라 쓴다. `src/main.js:153-156`의 위치값도 5등분 가정일 뿐 실제 얼굴 위치 검출이나 개별 아트 보장이 없다.

영향:
- 캐릭터 선택 화면과 대화 화면에서 캐릭터별 정체성이 약하다.
- 표정 변화, 의상 변화, 호감도 단계별 포즈, 이벤트 CG를 구현할 수 없다.
- 합성 일러스트 구도가 바뀌면 잘리는 위치가 틀어질 가능성이 높다.

필요 조치:
- 캐릭터별 기본 전신/반신 일러스트 5종 이상.
- 캐릭터별 표정 스프라이트 또는 최소 표정 3종.
- 대화용 초상화, 선택 화면용 카드, 엔딩 CG를 분리 저장.

### P0-2. 튜토리얼이 없다

`src/main.js:622-634`의 화면 렌더러에는 `tutorial` 화면이 없다. `src/data/content.js:122`에 "튜토리얼 던전"이라는 장소명만 있을 뿐, 에너지/카드 비용/방어/턴 종료/보상/루트 선택을 단계별로 알려주는 시스템은 없다.

영향:
- 첫 플레이어가 덱빌딩 규칙을 스스로 추측해야 한다.
- 카드 텍스트가 영어라 튜토리얼 부재의 손상이 더 크다.
- 15세 대상 캐주얼 플레이어에게 진입 장벽이 높다.

필요 조치:
- 첫 회차 강제 튜토리얼 전투.
- 카드 1장 사용, 턴 종료, 방어, 드로우, 보상 선택을 단계별 오버레이로 안내.
- 튜토리얼 완료 여부를 저장해 재시작 시 건너뛰기 제공.

### P0-3. 한영 혼용이 전반적이다

카드 데이터는 거의 전부 영어다. 예시는 `src/data/cards.js:16-24`의 `Warm Wave`, `Deal 8 charm.`, `src/data/cards.js:26-33`의 `Quick Think`, `Deal 7 wit. A small chance to draw.`이다. 영향:
- 한국어 게임으로서 완성도가 크게 떨어진다.
- 카드 효과 이해가 느려지고, 플레이 중 피로도가 높아진다.
- 미연시 톤과 시스템 UI가 서로 다른 제품처럼 보인다.

필요 조치:

- 내부 id/tag는 영어 유지 가능하나 사용자에게 보이는 `name`, `text`, `rarity`, `family`, `route`, `stat`은 한국어화.
- 용어집: 매력, 기지, 집중, 방어, 템포, 드로우, 체력, 비용 등.

### P0-4. 애니메이션 이펙트와 전투 피드백이 없다

CSS에는 `@keyframes`가 없고, 검색 기준으로 `animation` 규칙도 없다. `src/styles.css:261-264`의 hover `transform` 외에는 카드 사용, 피해, 회복, 보상, 화면 전환, 엔딩 해금 연출이 없다.

영향:
- 전투에서 카드가 "사용됐다"는 감각이 약하다.
- 덱빌딩 로그라이크의 손맛이 없다.
- 미연시 이벤트 전환도 정적인 텍스트 교체에 머문다.

필요 조치:
- 카드 사용 시 카드 플라이/소멸/타격 이펙트.
- 피해 숫자, 회복 숫자, 방어막 증가, 적 체력 감소 애니메이션.
- 대화 등장/퇴장, 선택지 등장, 엔딩 해금 전환 연출.

### P0-5. 문서가 실제 구현보다 기능을 과장한다

필요 조치:

## 콘텐츠 및 서사 문제

### P1-1. 스토리 125페이지는 실제 125개의 고유 장면이 아니다

`src/data/content.js:155-191`의 5개 `storyTemplates`를 캐릭터 5명, 액트 5개에 조합해 `src/data/content.js:252-280`에서 125개 페이지로 만든다. 즉, 수량은 125이지만 대부분 같은 문장 구조에 이름/장소만 바뀐다.

영향:
- 사용자가 몇 번만 진행해도 반복감을 느낀다.
- "최소 100페이지"라는 요구를 형식적으로만 만족한다.
- 루트별 감정선, 갈등, 선택 결과가 얕다.

필요 조치:
- 캐릭터별 고유 에피소드 20개 이상.
- 액트별 갈등 구조, 중간 반전, 고백 전야, 엔딩 분기를 직접 작성.
- 템플릿은 반복 이벤트용으로만 제한.

### P1-2. 대사 125페이지도 5개 템플릿 반복이다

`src/data/content.js:193-224`의 5개 `dialogueTemplates`를 조합해 `src/data/content.js:283-312`에서 125개 대사로 만든다. 말투가 캐릭터마다 충분히 다르지 않고, 상황별 반응도 얕다.

영향:
- 츤데레, 백치미, 테토녀 스타일 등 캐릭터성이 표면 설명에 머문다.
- 호감도 단계별 변화가 없다.
- 선택지 결과가 감정적으로 누적되는 느낌이 약하다.

필요 조치:
- 캐릭터별 말투 규칙표와 금칙어.
- 호감도 0/5/10/15/18 이상 단계별 대사.
- 승리/패배/상점/휴식/보스/엔딩 직전 전용 대사.

### P1-3. 캐릭터 설정 일부가 요청과 어긋나거나 부족하다

요청은 특정 여성 캐릭터 유형을 요구했지만, 현재 5명 중 `src/data/content.js:98-114`의 오준은 남성 캐릭터다. 이것이 나쁜 선택은 아니지만, 사용자의 명시 기대와는 어긋날 수 있다. 또한 유하나는 `archetype`에 `curvy cheerful airhead`가 있으나 `src/data/content.js:44-45`의 실제 소개문에는 외형적 특성과 백치미가 충분히 드러나지 않는다.

영향:
- 요청한 캐릭터 판타지와 실제 노출 정보가 다르다.
- 캐릭터 선택 화면에서 archetype 문자열 일부가 영어 tag로 노출되어 몰입감이 떨어진다.

필요 조치:
- 요청 캐릭터 3명은 한국어 설정과 전용 이벤트로 확실히 분리.
- 추가 캐릭터 성별/역할은 기획 문서에 의도 명시.
- README의 성적 콘텐츠 허용 기준을 따르되, 모든 로맨스 대상은 성인으로 유지하고 상호 존중을 해치는 묘사는 배제한다.

### P1-4. 루트 전용 보상과 hook이 실제 게임에 연결되지 않는다

`src/data/content.js:234-237`은 `unlocks`를 만들지만 `src/main.js`에서 `hook.unlocks`를 소비하지 않는다. `cardRewards`도 데이터에만 있고 실제 카드 카탈로그와 연결되지 않는다.

영향:
- 캐릭터 루트를 타도 덱빌딩 전략이 크게 달라지지 않는다.
- 미연시 선택과 덱 성장이 결합되지 않는다.

필요 조치:
- hook unlocks를 실제 카드 보상 풀, 유물, 이벤트 해금에 연결.
- 캐릭터별 전용 카드/유물 데이터 추가.
- 특정 선택지로 카드 강화/제거/전용 이벤트 개방.

### P1-5. 엔딩 조건 메타데이터가 실제 판정과 맞지 않는다

`src/data/content.js:323-382`의 `requiredAffinity`는 선언되어 있지만, 실제 판정은 `src/main.js:594-612`의 단순 로직만 사용한다. `ending-solo-master-builder`의 `clearedRuns: 3`, `romanceLocked: false` 조건은 적용되지 않는다. 실패 엔딩 메타의 `failedBoss`도 실제 엔딩 선택에서는 패배 시 고정 엔딩으로만 쓰인다.

영향:
- 엔딩 설명과 실제 해금 조건이 다르다.
- 멀티 엔딩이라고 보기에는 분기 깊이가 얕다.
- 갤러리 수집 목표가 설계대로 작동하지 않는다.

필요 조치:
- 엔딩 조건 평가기를 별도로 구현.
- `requiredAffinity`, 회차 수, 보스 승리, 기억 조각, 핵심 선택 플래그를 모두 판정.
- 엔딩별 힌트 UI 추가.

## 게임 시스템 문제

### P1-6. 전투가 단일 적, 단일 대상, 단순 수치 계산에 머문다

전투는 `src/main.js:438`에서 적 HP를 직접 깎고, `src/main.js:473-475`에서 적 공격을 단순 피해로 처리한다. 상태 이상, 복수 적, 대상 선택, 방어 지속, 적 패턴 예고, 카드 보존, 소멸/생성, 강화 카드 같은 장르 핵심 요소가 없다.

영향:
- 덱빌딩 선택의 전략성이 낮다.
- 카드 이름과 효과만 많고 실제 조합의 체감이 약하다.
- 1시간 이상 플레이 시 반복 피로가 커진다.

필요 조치:
- 적 의도 시스템: 공격/방어/버프/디버프 예고.
- 카드 키워드: 보존, 소멸, 생성, 취약, 집중, 연소, 앙코르 등.
- 복수 적 또는 보스 페이즈.
- 카드 강화/제거/변환.

### P1-7. 맵이 로그라이크 경로 선택처럼 작동하지 않는다

`src/main.js:759-765`는 매 노드마다 사실상 2~3개 선택지만 새로 보여준다. 분기 경로, 되돌릴 수 없는 선택, 위험/보상 표시, 장기 경로 계획이 없다.

영향:
- Slay the Spire류 로그라이크 맵의 전략성이 없다.
- "위험한 경로일수록 강한 보상"이라는 매뉴얼 설명과 다르다.

필요 조치:
- 주차 시작 시 노드 그래프 생성.
- 엘리트/이벤트/휴식/상점/보스 경로 연결 표시.
- 위험도와 보상 예측치 표시.

### P1-8. 상점이 선택지가 아니라 자동 처리다

`src/main.js:354-359`에서 상점은 골드 12 차감, 체력 12 회복, 랜덤 카드 추가를 즉시 수행한다. 구매 목록, 카드 제거, 강화, 유물 구매가 없다.

영향:
- 플레이어 결정권이 없다.
- 골드 자원 관리가 의미가 약하다.
- 상점 이벤트의 재미가 사라진다.

필요 조치:
- 상점 화면 추가.
- 카드 3장, 유물 2개, 제거/강화 서비스 제공.
- 골드 부족 시 비활성화와 설명.

### P1-9. 보상도 카드 1장 선택 외 깊이가 낮다

`src/main.js:554-563`은 카드 1장 선택 또는 건너뛰기만 처리하고, 유물이 있으면 선택 여부와 무관하게 자동 지급된다. 보상 카드 강화, 회복, 골드, 제거, 루트 전용 보상 선택이 없다.

영향:
- 전투 후 선택이 단조롭다.
- 유물 획득의 의식감과 연출이 없다.

필요 조치:
- 카드/유물/회복/골드 보상 중 선택 구조.
- 루트 호감도에 따른 보상 보정.
- 보상 획득 연출.

### P1-10. 설정 UI가 실제 효과를 적용하지 않는다

`src/main.js:1092-1100`은 `textSize`, `reducedMotion` 값을 저장하지만, CSS나 렌더링 로직에서 이를 사용하지 않는다. `src/main.js:999-1023`의 설정은 표시만 바꾸는 수준이다.

영향:
- 접근성 옵션이 작동하지 않는 허위 기능이 된다.
- "움직임 줄이기"는 애니메이션이 없어서 의미가 없다.

필요 조치:
- 루트 클래스 또는 CSS 변수로 텍스트 크기 적용.
- 애니메이션 추가 후 `prefers-reduced-motion`와 사용자 설정 반영.

### P1-11. 저장 데이터 안정성이 약하다

`src/main.js:86-103`은 JSON 파싱 실패만 처리하고, 저장 데이터 버전/스키마 검증/마이그레이션이 없다. 카드 id가 변경되거나 데이터 구조가 바뀌면 로드 후 런타임 오류가 날 수 있다.

영향:
- 업데이트 후 기존 저장 데이터가 깨질 수 있다.
- GitHub Pages 배포 이후 버전업에 취약하다.

필요 조치:
- save schema version 추가.
- 로드 시 deck/relic/character/ending id 검증.
- 깨진 save는 복구 선택지 제공.

## UI/UX 및 접근성 문제

### P2-1. 모바일 전투 UI는 카드가 가로 스크롤에 의존한다

`src/styles.css:473-476`의 `.hand`는 overflow-x 스크롤이고, 카드 폭은 `src/styles.css:479-482`, 모바일에서도 `src/styles.css:682-684`로 큰 편이다. 장시간 플레이에서 손패 탐색 피로가 크다.

필요 조치:
- 모바일 카드 축약 모드.
- 카드 확대 보기.
- 손패 정렬/필터/선택 후 확인 단계.

### P2-2. 접근성 속성이 부족하다

대부분의 카드/노드 버튼은 `aria-label`, 상태 설명, 라이브 리전이 없다. 전투 로그도 `aria-live`가 없어 스크린리더 사용자에게 변화가 전달되지 않는다.

필요 조치:
- HP/에너지/전투 로그에 `aria-live`.
- 카드 버튼에 비용, 효과, 사용 가능 여부를 포함한 `aria-label`.
- 키보드 포커스 순서 검증.

### P2-3. 버튼/카드 상태 피드백이 약하다

비활성 카드는 `src/styles.css:495-497`에서 opacity만 낮춘다. 왜 사용할 수 없는지, 에너지가 얼마나 부족한지, 사용 시 결과가 무엇인지 미리 알려주지 않는다.

필요 조치:
- 카드 hover/focus 시 예상 피해/방어/회복 표시.
- 에너지 부족 사유 툴팁.
- 사용 가능 카드 강조.

### P2-4. 갤러리와 엔딩 화면이 빈약하다

`src/main.js:972-995`의 갤러리는 잠금/해금 텍스트만 보여준다. 엔딩 CG, 해금 조건 힌트, 다시 보기, 캐릭터별 필터가 없다.

필요 조치:
- 엔딩 CG와 회상 장면.
- 해금 조건 힌트.
- 캐릭터별/친구 엔딩/솔로 엔딩 필터.

### P2-5. 오디오가 전혀 없다

문서에서는 음량 설정을 언급하지만 실제 오디오 파일, 오디오 매니저, 효과음, BGM이 없다.

필요 조치:
- 타이틀/맵/전투/엔딩 BGM.
- 카드 사용, 피해, 회복, 보상, 선택지 효과음.
- 음량/음소거 설정.

## 아트/에셋 파이프라인 문제

### P2-6. 에셋 수와 용도가 부족하다

현재 이미지 파일은 6개뿐이다. `assets/images/title-bg.png`, `characters.png`, `battle-bg.png`, `card-sheet.png`, `map-bg.png`, `ending-bg.png`가 전부다. 캐릭터별, 카드별, 적별, UI별, 이펙트별 에셋이 없다.

필요 조치:
- 캐릭터 5명 x 표정 3종.
- 적/보스 11종 이상.
- 카드 패밀리별 프레임, 희귀도 프레임.
- 이펙트 스프라이트 또는 CSS/Canvas 이펙트.

### P2-7. 이미지가 압축되지 않아 로딩 비용이 크다

각 PNG가 약 2.5MB~3.1MB이며 총 16MB 이상이다. 정적 웹게임 첫 로딩으로는 부담스럽다.

필요 조치:
- WebP/AVIF 변환.
- 썸네일/원본 분리.
- 지연 로딩과 preload 재검토.

### P2-8. 카드 아트는 카드별 고유 아트가 아니다

`src/main.js:159-163`은 카드 id hash로 16칸 스프라이트 중 하나를 배정한다. 카드가 55장인데 스프라이트는 16칸이라 다수 카드가 같은 그림을 반복한다.

필요 조치:
- 최소 카드 패밀리별 8~12종, 핵심 카드 고유 아트.
- 카드 id와 아트 좌표를 명시 매핑.
- 고유 보스/루트 카드에는 전용 일러스트.

## 밸런스 및 검증 문제

### P2-9. 밸런스 시뮬레이션이 실제 브라우저 전투와 다르다

`tools/balance-sim.mjs`는 자체 평가식으로 전투력을 계산하고, 브라우저 전투의 손패/카드 사용/턴 종료와 완전히 같지 않다. 따라서 57.0% 승률은 실제 플레이 난이도를 보장하지 않는다.

필요 조치:
- 전투 계산 코어를 공용 모듈로 분리.
- 시뮬레이터와 브라우저가 같은 전투 엔진을 사용.
- 결정론적 seed로 회귀 테스트 작성.

### P2-10. 테스트가 "존재 여부" 중심이다

`tools/smoke-test.mjs`는 파일 존재, 데이터 개수, 참조 일부만 확인한다. 튜토리얼 부재, 한영 혼용, 설정 미적용, 문서 불일치, UI 깨짐은 잡지 못한다.

필요 조치:
- Playwright 기반 핵심 플로우 테스트.
- 현지화 누락 검사.
- 문서-구현 기능 매트릭스 검사.
- 모바일 스크린샷 회귀 테스트.

### P2-11. 실제 1시간 플레이 검증이 없다

README와 디자인 문서는 78분 풀런을 말하지만, 이는 `levelCurve.estimatedMinutes` 합산과 시뮬레이션 추정치다. 실제 사람이 플레이했을 때 1시간 이상이 되는지 검증 로그가 없다.

필요 조치:
- QA 플레이 세션 기록.
- 평균 텍스트 읽기 시간, 전투 시간, 선택 시간 측정.
- 빠른 클릭으로 20분 이내 종료되는지 확인.

## 운영/배포 문제

### P3-1. GitHub Pages 배포 설정이 없다

게임은 GitHub에 푸시되어 있지만, `.github/workflows`나 Pages 설정 파일이 없다. 사용자가 URL로 즉시 플레이할 수 있다는 보장은 없다.

필요 조치:
- GitHub Pages 배포 워크플로.
- 정적 파일 캐시 정책.
- 배포 후 URL README에 기재.

### P3-2. 빌드/패키징 체계가 없다

`package.json:7-8`은 Python HTTP 서버와 check만 제공한다. npm 의존성은 없고 빌드도 없어서 간단하지만, 압축/최적화/해시 파일명/배포 전 검증 흐름이 없다.

필요 조치:
- 정적 빌드 스크립트 또는 배포 스크립트.
- 이미지 최적화 스크립트.
- 릴리스 태그와 changelog.

### P3-3. 소스 모듈이 하나의 큰 파일에 몰려 있다

`src/main.js`는 상태, 렌더링, 전투, 저장, 입력, 엔딩 판정을 모두 포함한다. 유지보수와 테스트가 어렵다.

필요 조치:
- `state.js`, `combat.js`, `renderers/`, `save.js`, `endings.js`, `i18n.js`로 분리.
- 전투 엔진은 순수 함수화.

## 우선순위별 개선 로드맵

- 모든 표시 문자열 한국어화.

- 매뉴얼에서 미구현 기능 삭제 또는 실제 구현.

### 2차 패치: "미연시와 덱빌딩 결합"

## 결론

현재 게임은 "시작해서 전투하고 엔딩을 볼 수 있는 프로토타입"은 맞다. 그러나 "완벽한 게임" 또는 "즉시 출시 가능한 1시간 이상 분량의 미연시 덱빌딩 로그라이크"로 보기는 어렵다. 가장 먼저 해결해야 할 것은 개별 캐릭터 일러스트, 튜토리얼, 한국어화, 애니메이션 피드백, 문서-구현 불일치다. 그 다음에야 캐릭터 루트와 덱빌딩이 실제로 결합된 게임으로 다듬을 수 있다.
